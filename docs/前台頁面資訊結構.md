# 前台頁面資訊結構（LLM 產出）

> 線框前的「資訊結構」：層次、區塊、優先序、第一眼焦點。
> 由 LLM 依 `docs/bpmn/*.md` 規格推導，已過自我挑戰一輪。**未比對已建置頁面。**
> 每頁格式：A 目標｜B 第一眼焦點｜C 區塊清單(優先序)｜D 狀態差異｜E 故意省略 / 需補。

---

## 1. VenueList（場地列表）｜來源 04

**A. 頁面目標**
讓會員在最短時間內篩出「現在可租、條件符合我」的場地。

**B. 第一眼焦點**
篩選/搜尋列 + 場地卡片牆。會員不是來逛的，是來「找符合條件的」——先給篩選，再給結果。

**C. 區塊清單（優先序）**
1. [篩選/搜尋] 依模式（整租/時段/包月）、可租狀態、價格、是否需文件篩選
2. [結果計數＋排序] 「符合 N 筆」＋排序（價格/可租近期）
3. [場地卡片] 每張：名稱、縮圖、可租模式標籤、價格起、是否需文件徽章、可租狀態
4. [分頁/載入更多]
5. [空狀態] (次要) 篩無結果時：放寬建議，而非空白

**D. 狀態差異**
- 場地「暫不可租 / 規則更新中」：卡片置灰並標原因，不可點進預約（呼應 04 規則發布即時生效）
- 篩選後 0 筆：第一眼焦點改為「放寬條件」提示

**E. 故意省略 / 需補**
- 卡片不放完整規則全文，只放決策用的最小資訊（模式、價格起、是否需文件），細節留詳情頁。
- 需補：可租狀態的判定來源、價格「起」的計算規則（來自 04 規則集）。

---

## 2. VenueDetail（場地詳情）｜來源 04

**A. 頁面目標**
讓會員判斷「這場地合不合用、怎麼租、要先準備什麼」，並順勢進入預約。

**B. 第一眼焦點**
場地識別（名稱／主圖／一句定位）＋ 一個明確的「立即預約」CTA。判斷與行動入口要同框。

**C. 區塊清單（優先序）**
1. [場地識別] 名稱、主圖、容量/地點等定位資訊
2. [可租模式與限制] 此場地允許的模式與使用限制（04 規則集）
3. [時段與價格] 各模式對應時段與費率
4. [文件需求與附加規則] 需備文件、押金/保證金、注意事項
5. [立即預約] 主CTA → VenueBooking（建議隨頁面吸附）
6. [圖片/設備] (次要/可收合) 更多照片、設備清單
7. [常見問題/條款] (次要/可收合)

**D. 狀態差異**
- 需文件的場地：第4塊上移並加重，CTA 文案「預約並準備文件」
- 暫不可租：CTA 置灰並說明原因/開放時間

**E. 故意省略 / 需補**
- 不在詳情頁做日期可用性即時查詢的重邏輯——交給預約頁，詳情頁只給「怎麼租」的認知。
- 需補：模式↔時段↔價格的資料結構，來自 04 發布的規則集。

## 3. VenueBooking（預約申請）｜來源 01、04

**A. 頁面目標**
讓會員在「該場地已發布規則」框架內，選定模式與時段、填好資料並安心送出，建立 reserved。

**B. 第一眼焦點**
「租借模式選擇」。模式決定後面所有欄位（時段、價格、是否需文件）；模式沒選，其餘都是噪音。

**C. 區塊清單（優先序）**
1. [場地與模式] 只顯示該場地允許的模式
2. [時段與費用] 依模式展開可選時段，即時算金額（04 規則）
3. [申請資料] 申請人/單位、用途等必填
4. [文件需求提示] 若 requireDocuments=是：明示送出後要上傳什麼＋期限
5. [送出動作] 主CTA：送出申請；次CTA：返回
6. [後續流程預告] (次要/可收合) 送出後：reserved → 文件審核/繳費

**D. 狀態差異（依規則）**
- requireDocuments=是：第4塊加重，CTA 文案「送出並準備上傳文件」
- requireDocuments=否：隱藏第4塊，提示「送出後將產生繳費期限」

**E. 故意省略 / 需補**
- 不放退款政策全文（只放一句連結），避免淹沒當下決策。
- 此頁不顯示匯款/帳務資訊——那是 reserved 之後的階段。
- 需補：各模式「可選時段來源」與「金額計算規則」。

## 4. VenueConfirm（送出前確認）｜來源 01

**A. 頁面目標**
送出前讓會員核對所有選擇，把「送錯」攔在建立 reserved 之前。

**B. 第一眼焦點**
一份可掃讀的「申請摘要」。確認頁的價值在「讓人一眼抓出哪裡填錯」，不是再讓人填東西。

**C. 區塊清單（優先序）**
1. [申請摘要] 場地、模式、時段、金額、申請人——每項可「返回修改」
2. [後續義務提示] 是否需文件、文件/繳費期限預告（讓人知道送出後要做什麼）
3. [條款同意] 必要勾選項（取消/退款政策連結）
4. [確認送出] 主CTA：確認送出（→建立 reserved）；次CTA：返回修改

**D. 狀態差異**
- 需文件：第2塊明示「送出後 X 天內上傳，逾期將自動取消(cancelled_rejected)」
- 不需文件：第2塊明示「將產生繳費期限，逾期自動取消(cancelled_expired)」

**E. 故意省略 / 需補**
- 不重複完整場地介紹，只留決策必需的摘要。
- 需補：取消/退款政策文案來源。

## 5. MyBookings（我的預約列表）｜來源 01

**A. 頁面目標**
讓會員一眼看出每筆預約「現在卡在哪、要不要我動作」。

**B. 第一眼焦點**
「需我動作」的預約優先置頂。列表型頁面的焦點不是全部，是「現在輪到我做的那幾筆」。

**C. 區塊清單（優先序）**
1. [需我動作區] 待上傳文件 / 待補件 / 待繳費 的預約，置頂高亮＋倒數
2. [狀態篩選] 進行中 / 已成立 / 已取消
3. [預約列表] 每列：場地、時段、狀態徽章、期限倒數、單筆主行動入口
4. [空狀態/引導] (次要) 無預約時導向 VenueList

**D. 狀態差異（每列徽章與行動）**
- reserved＋待文件：徽章「待上傳文件」/ 行動=上傳
- 文件退件：徽章「需補件」(警示色) / 行動=補件
- pending_payment：徽章「待繳費」＋倒數 / 行動=繳費
- payment_review：徽章「核帳中」/ 無行動
- confirmed：徽章「已成立」/ 行動=查看憑證
- cancelled_expired / cancelled_rejected：徽章「已取消」＋原因 / 無行動

**E. 故意省略 / 需補**
- 列表只放「狀態＋下一步」，細節進 BookingDetail，避免每列資訊爆炸。
- 需補：期限資料、各狀態的徽章文案。

---

## 6. BookingDetail（預約詳情·訂單指揮中心）｜來源 01、02、05、06

**A. 頁面目標**
讓會員看懂這筆訂單現在的狀態、卡在哪、下一步該做什麼——一頁回答「我現在要不要動作」。

**B. 第一眼焦點**
「狀態橫幅 + 一句白話 + 下一步行動」。例：待文件→「請於 X 前上傳文件」；待繳費→「請於 X 前完成繳費」。

**C. 區塊清單（優先序）**
1. [狀態橫幅] 狀態白話化＋期限倒數（逾期類資訊置頂）
2. [下一步行動] 依狀態的主CTA（上傳/補件/提交匯款證明/補正/提出取消）
3. [待處理原因] 退件原因、核帳不一致原因——緊跟行動，不可埋深（高情緒資訊）
4. [訂單摘要] 場地、模式、時段、金額
5. [進度時間軸] reserved→審核→繳費→confirmed；取消線另顯示退款三段簽核(02)
6. [歷程紀錄] (次要/可收合) 每次狀態變更與時間
7. [次要操作] (次要/可收合) 下載憑證、聯絡客服

**D. 狀態差異（第一眼焦點 / 主CTA）**
| 狀態 | 焦點 | 主CTA |
|---|---|---|
| reserved＋待文件 | 上傳倒數 | 上傳文件 |
| 文件退件 | 退件原因 | 補件 |
| pending_payment | 繳費倒數 | 提交匯款資訊與證明 |
| payment_review | 「核帳中，無須動作」 | 無（防重複付款） |
| 核帳不一致 | 補正提示 | 補正匯款資料 |
| confirmed | 成立憑證 | 無；次CTA=申請取消 |
| cancellation_requested | 「取消審核中」 | 無 |
| cancelled＋退款中 | 退款進度(承辦→會計→出納) | 無 |
| cancelled_expired / cancelled_rejected | 終局說明＋原因 | 無 |

**E. 故意省略 / 需補**
- 終局狀態不放「再次預約」大按鈕搶注意力，只以次要操作出現。
- 後台簽核內部欄位不對會員顯示，只顯示「進度到哪」。
- 需補：各狀態的白話文案表、期限資料對應表。

## 7. MyProfile（我的帳戶 / 留存保證金）｜來源 03

**A. 頁面目標**
讓會員看到留存保證金餘額，並能正確發起保證金退款申請。

**B. 第一眼焦點**
「留存保證金餘額」＋「申請退款」入口。這頁對會員的核心價值是錢在哪、能不能拿回。

**C. 區塊清單（優先序）**
1. [保證金餘額] 目前留存金額（退款申請的金額上限依據）
2. [申請退款] 主CTA → 退款申請表
3. [退款申請表] 金額（不可超過餘額）、原因、銀行帳戶——含即時金額校驗
4. [帳戶基本資料] (次要/可收合) 個資/聯絡方式維護
5. [送出後預告] (次要) 「將進入後台簽核：承辦→會計→出納」，並導向 MyRefunds 追蹤

**D. 狀態差異**
- 餘額為 0：第2塊置灰，說明「目前無可退保證金」
- 金額超過餘額或資料不全：阻擋送出並就地提示（呼應 03 的資料完整＋金額合法閘道）

**E. 故意省略 / 需補**
- 不在此頁顯示退款進度細節——進度歸 MyRefunds，此頁只負責「發起」。
- 需補：保證金餘額資料來源、銀行帳戶欄位驗證規則。

---

## 8. MyRefunds（我的退款）｜來源 03、02

**A. 頁面目標**
讓會員追蹤每筆退款案件走到哪、最後結果是什麼。

**B. 第一眼焦點**
退款案件列表，每筆以「進度條 + 目前階段」呈現。會員最在意「我的錢處理到哪了」。

**C. 區塊清單（優先序）**
1. [退款案件列表] 每筆：類型(保證金/訂單取消)、金額、目前階段、申請日
2. [進度時間軸] 單筆展開：admin_review→accounting_review→cashier_processing→completed
3. [結果/原因] completed 顯示撥款資訊；rejected 顯示駁回原因（高情緒，需明顯）
4. [類型篩選] (次要) retained_deposit / booking_cancellation
5. [空狀態] (次要) 無退款時的說明

**D. 狀態差異（每筆呈現）**
- 進行中：進度條停在當前階段，文案「審核中／撥款處理中」
- completed：綠色完結＋撥款資訊
- rejected：警示色＋駁回原因置前，提供「聯絡客服/重新申請」次要入口

**E. 故意省略 / 需補**
- 不顯示後台各角色的內部審核欄位，只顯示對會員有意義的「階段＋結果」。
- 需補：兩種退款類型的資料合流方式（03 保證金、02 訂單取消）、rejected 原因文案來源。

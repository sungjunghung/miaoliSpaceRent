## 行為準則

### 1. 實作前先思考

**不要假設。不要隱藏疑惑。主動提出取捨。**

實作前：
- 明確說出假設前提。有疑問就問。
- 有多種解讀方式時，逐一列出，不要自行暗中決定。
- 有更簡單的做法時，說出來。有理由反對時，提出來。
- 有不清楚的地方，停下來，說明哪裡不清楚，然後問。

### 2. 以簡為先

**用最少的程式碼解決問題，不做推測性的實作。**

- 不做超出需求的功能。
- 單次使用的程式碼不需要抽象化。
- 沒有被要求「彈性」或「可設定性」就不要做。
- 不可能發生的情境不需要錯誤處理。
- 如果寫了 200 行但 50 行就夠，重寫。

自問：「資深工程師會說這過於複雜嗎？」如果會，就簡化。

### 3. 精準修改

**只動必須動的地方，只清理自己造成的問題。**

修改既有程式碼時：
- 不要順手「改善」周邊的程式碼、註解或排版。
- 不要重構沒有壞掉的東西。
- 維持既有風格，即使你會用不同方式寫。
- 發現不相關的廢棄程式碼，提出來，不要自行刪除。

當你的修改造成孤立程式碼時：
- 移除因你的修改而不再使用的 import、變數、函式。
- 不要移除原本就存在的廢棄程式碼，除非被要求。

驗證標準：每一行改動都要能直接對應到使用者的需求。

### 4. 目標導向執行

**定義完成標準，確認達成後才算完成。**

將任務轉化為可驗證的目標：
- 「新增驗證」→「為無效輸入寫測試，然後讓測試通過」
- 「修復 bug」→「寫出能重現問題的測試，然後讓測試通過」
- 「重構 X」→「確認重構前後測試都通過」

多步驟任務時，先說明計畫：
```
1. [步驟] → 確認：[檢查方式]
2. [步驟] → 確認：[檢查方式]
3. [步驟] → 確認：[檢查方式]
```

明確的完成標準讓你能獨立循環執行。模糊的標準（「讓它能用」）會導致不斷確認。

**這些原則有效的跡象：** diff 中不必要的修改變少、因過度複雜而重寫的次數減少、釐清問題的提問發生在實作之前而非出錯之後。

## 架構

**Vue 3 + Vite** 原型，空間租借平台。

分為兩個區塊：

- **前台（portal）** — 給一般使用者：場地、文章、會員、登入
- **後台（admin）** — 給管理員：`/admin/*` 路由

### 目錄與資料流規範

**資料存取：`mocks/` 只允許被 `services/` import。**
views、components、layouts、stores 一律透過 `services/` 或 stores 取得資料，不得直接 import mock JSON。接 API 時只需改寫 services 內部實作。

**元件歸位依使用範圍決定：**

- 被兩個以上頁面使用 → `components/{admin,portal}/`
- 只有一個頁面使用 → 該頁面自己的 `components/`（如 `views/admin/calendar/components/`）
- 複合元件的子區塊跟著主元件放同一層，不再細分

**命名慣例：**

- SFC 檔名用 PascalCase；views 內的路由頁採「資料夾 + index/detail/edit.vue」（admin 區）
- route name：portal 不加前綴（`home`、`login`、`venue-list`），admin 一律 `admin-` 前綴
- 跨目錄 import 一律用 `@/` alias，同資料夾用 `./`
- `composables/` 只放有響應式狀態的 `useXxx()`；純函式與常數表放 `utils/`
- 圖片等靜態資源檔名用英文

### 樣式

避免在 JS/TS 中動態拼接 class string（例如字串相加或 template literal），Tailwind 的 purge 掃描靜態原始碼，動態產生的 class 可能在打包時被移除。請改用完整 class string 的物件對應：

```js
const cls =
  { open: "badge-success", draft: "badge-ghost" }[status] ?? "badge-ghost";
```

**CSS 規範：**

- 以 DaisyUI 為基礎元件
- 再疊加必要的 TailwindCSS utility class
- 採用 BEM（Block\_\_Element--Modifier）命名規則
- CSS 命名需具語意，但避免過度語意化
- 若 DaisyUI 已提供對應元件，直接使用即可，不需重新語意化
- 僅在有延伸需求時，才額外建立語意化 CSS 並進行疊加

## 指令

> 專案位於 `frontend/`，需先 `cd frontend` 再執行以下指令。

```bash
npm run dev       # 啟動開發伺服器 http://localhost:5173
npm run build     # 型別檢查（vue-tsc）並建置正式環境
npm run preview   # 預覽 build 產物
```

> `npm` 指令需直接在終端機執行，Claude Code 的 Bash 工具與 nvm 不相容。
